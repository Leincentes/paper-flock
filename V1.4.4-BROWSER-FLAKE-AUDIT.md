# Paper Flock v1.4.4 — Browser Flake Audit

## Incident

The GitHub browser-quality job completed with:

- 58 passed
- 6 skipped
- 2 flaky
- exit code 1

The flaky tests were:

1. desktop Chromium — Settings traps focus and restores it after closing
2. mobile Chromium — game starts without uncaught page errors

## Root causes

### 1. First-install service-worker reload race

`service-worker.js` calls `clients.claim()` during activation. The player runtime
previously reloaded the page for every `controllerchange` event. On a clean
browser profile, the first service-worker activation can therefore claim the
already-open page and trigger an unsolicited reload while startup assertions
are running.

The race is more visible under mobile emulation because startup and layout work
take longer. A retry often passes because the service worker is already active.

### 2. Two independent dialog focus managers

`settings-ui.js` and `journal-ui.js` each implemented their own Escape handler,
Tab trap, initial focus, and focus restoration. `accessibility-ui.js` also
implements a central modal manager for every `aria-modal` dialog.

The independent handlers schedule competing `requestAnimationFrame` focus
operations. Depending on observer and animation-frame ordering, focus can be
restored by one controller and then changed by the other.

### 3. The E2E setup did not wait for the complete player runtime

`openReturningPlayerGame()` waited for the board to be visible but not for the
`ui-ready` marker or dynamically injected Settings control. This allowed tests
to begin while final startup work was still pending.

## Resolution

- Reload on `controllerchange` only after the player explicitly accepts an
  available update.
- Delegate Settings and Journal focus trapping, Escape handling, and focus
  restoration to the single accessibility dialog controller.
- Make the E2E setup wait for `html.ui-ready`, the board, and the Settings
  control before interacting.
- Explicitly fail CI when a test is flaky, so retries remain diagnostic rather
  than silently approving an unstable build.
- Add unit-level source-contract tests for service-worker reload gating,
  centralized dialog ownership, and E2E readiness.
